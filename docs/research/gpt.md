# Web Speed Hackathon と類似パフォーマンスコンテスト調査報告

## エグゼクティブサマリ

過去の「Web Speed Hackathon」は、重いWebアプリを短時間で“壊さずに”高速化し、主に **Lighthouse 系指標（近年は v10 の重み付け）** と **ユーザーフロー指標（TBT/INP）**、開催回によっては **動画再生の実測** まで含めてスコア化する、フロントエンド寄りのパフォーマンス競技として設計されています。たとえば 2025 は「ページ表示（900）＋ページ操作（200）＋動画再生（100）=1200点」で、表示スコアが一定以上のときだけ後段の採点へ進む“ゲート”があるなど、スコア計測の“仕様”を理解して最適化順序を組むことが勝敗に直結します。citeturn21view0turn21view3

最大の落とし穴は「速くしたがレギュレーション落ち」です。複数年で「Chrome最新版での著しい機能落ち・デザイン差異」や VRT（Visual Regression Test）・手動チェック項目の不合格が順位除外条件として明記され、2024 は Service Worker 登録や初期化 API など“採点が成立するための要件”も含まれます。citeturn20view1turn21view1turn22view1 参加記でも、最終的にレギュレーション違反で順位対象外になった例が複数報告されています。citeturn18search4turn18search0turn18search10

実務的な攻略の要点は、(1) **採点仕様と重み** を先に固定し、(2) **ローカルで再現可能な計測ループ**（スコアリングツール／Lighthouse CI／リグレッションテスト）を作り、(3) **高インパクト領域を優先**（メインスレッドのブロッキング、LCP要素の優先度、レイアウトシフト、バンドル肥大、キャッシュ／Service Worker など）して反復することです。Lighthouse の Performance スコアが「各メトリクスの重み付き平均」であること、v10 の重み（特に TBT/LCP/CLS）を押さえるのが近道です。citeturn11search4turn14search14

競技中に参照しやすい「一次資料（過去問題・採点・レギュレーション）」「主要ツール公式ドキュメント」「チュートリアル／チェックリスト」「実装例（GitHub・参加記・デモ動画）」を、開催年・問題形式・手法・ツール・スコア/ベンチ結果の観点で索引化しました。以下の表は **URL（必須）＋要約（2〜4文）** を満たし、未確認/明記なしは「未指定」としています。

## 調査の前提と読み方

Web Speed Hackathon は開催年により **スコア配点・対象ページ・採点手順・使用ツール（Lighthouse のバージョン相当の重み）** が変化します。2020 は Lighthouse v6 を前提とした相対スコア加点、2021〜2022 は v8 系の重み参照、2023〜2025 は v10 系の重み（＋ユーザーフローに INP）へ移行しています。citeturn28view0turn23view0turn20view3turn21view0

また、計測は大きく「ラボ計測（合成環境、Lighthouse等）」と「フィールド計測（実ユーザー、CrUX等）」に分かれ、両者は一致しないことがあります。コンテストでは通常ラボ寄りの採点が中心になるため、まずは採点ループをラボ計測で安定させ、そのうえで設計上の“本質的遅さ”を DevTools / プロファイラで潰す、という順序が実務的です。citeturn12search16turn12search1turn12search9

## Web Speed Hackathon 公式アーカイブ

### 年ごとの競技仕様サマリ

- 2020（Online Vol.1）: ブログ風アプリ「Amida Blog」。Lighthouse v6 を使い、Performance Score（0–100）に相対スコアを加点し、複数ページ合算で競う。citeturn28view0turn4view0  
- 2021（mini）: SNS風「CAwitter」。Lighthouse v8 重み参照で複数ページ合算、VRT を含む“壊さない”条件が明記される。citeturn25view0turn26view0  
- 2022（Public/一般向け）: ベッティング風「CyberTicket」。Lighthouse v8 重み参照で複数ページ合算、リーダーボードは CSV で全件公開。citeturn23view0turn23view2turn24view0  
- 2023: EC風「買えるオーガニック」。ページランディング＋ユーザーフロー（TBT/INP）を別枠で採点し、チェックリスト（手動観点）も具体化。citeturn22view2turn20view3turn22view1  
- 2024: 漫画サイト「Cyber TOON」。ページ4枚＋複数ユーザーフロー、採点成立要件として Service Worker 登録や初期化 API が入る。citeturn20view0turn20view1turn31view0  
- 2025: 動画配信「AREMA」。ページ表示（900）・操作（200）・動画再生（100）で 1200 点、動画は Puppeteer による“再生開始までの時間”を式でスコア化。citeturn21view0turn21view3

## 実践ワークフローとチェックリスト

### 競技中の“動き方”の推奨テンプレ

Web Speed Hackathon 2026 の公式スケジュール例では、初日にリポジトリ公開→計測更新、2日目に競技終了→解説→順位確定、という流れが提示されています。citeturn30view3 これを前提に、過去回のレギュレーション（VRT/手動チェック/アクセス維持/初期化APIなど）を満たしつつ反復する流れを、実務向けに落としたのが以下です。citeturn21view1turn22view1turn20view1

```mermaid
flowchart TD
  A[採点仕様を読む\nscoring/regulation/checklist] --> B[計測環境を固定\n(ローカル計測 or CI計測)]
  B --> C[ベースライン計測\nページ別/フロー別に分解]
  C --> D{どの点が落ちている？\nLCP vs TBT/INP vs CLS vs その他}
  D -->|LCP| E[主要LCP要素を特定\n(画像/フォント/SSR/TTFB/優先度)]
  D -->|TBT/INP| F[長タスクを特定\n(メインスレッド/再レンダ/イベント処理)]
  D -->|CLS| G[ズレ原因を特定\n(寸法/挿入/フォント/アニメ)]
  E --> H[改善を実装]
  F --> H
  G --> H
  H --> I[回帰チェック\nVRT/E2E/手動チェック]
  I -->|NG| J[差分を戻す/条件付きで緩和]
  I -->|OK| K[再計測→スコア更新]
  K --> C
```

### 2日間タイムライン例

公式の例では、初日10:00に全体説明、10:30にリポジトリ公開/開始、19:00に1日目終了、2日目10:00再開、18:30競技終了、以降解説と順位確定…という形です。citeturn30view3  
競技設計上は「早い段階で採点が通る状態（初期化API、SW登録、テストID維持）を作る」「終盤はデグレ防止とレギュレーション順守に寄せる」ほど、失格リスクを下げやすいです。citeturn20view1turn22view1turn18search4
